三款 Agent 产品架构深度拆解

系列文章 2/4
Agent 系列 Allen | allen00.top

三款 Agent 产品架构深度拆解：Claude Code、Codex、OpenClaw

系列文章（2/4）| 作者：Allen | allen00.top

上一篇我们聊了 Agent 是什么，以及三款产品的基本面貌。这一篇往深了走，拆开它们的引擎盖，看看里面到底是怎么运转的。

我不会贴源码，但会把架构讲清楚。如果你是产品经理，理解这些架构差异能帮你做出更好的产品决策；如果你是开发者，这些设计思路可以直接借鉴。

一、Agent 的通用架构：先画一张底图

不管是哪款 Agent 产品，底层都遵循同一个模式，就是 Agent Loop（智能体循环）。

用户给出目标
    ↓
┌─────────────────────┐
│  收集上下文（Gather） │
│  ↓                   │
│  模型推理（Think）    │
│  ↓                   │
│  执行动作（Act）      │
│  ↓                   │
│  检查结果（Verify）   │
│  ↓                   │
│  是否完成？           │
│  否 → 回到收集上下文  │
│  是 → 返回结果给用户  │
└─────────────────────┘

这个循环是所有 Agent 的心脏。据我观察，区别在于：每个产品怎么实现这个循环、循环里塞了什么东西、循环跑在哪里。

二、Claude Code：极致专注的单线程架构

2.1 整体架构

我拆解下来发现，Claude Code 的架构哲学可以用四个字概括：简单直接。

一个终端进程，一个 Agent Loop，一个上下文窗口。所有事情都在这一个进程里完成。

用户终端
  ↓
CLI 进程（Node.js）
  ├── Agent Loop 引擎
  ├── 工具执行层（文件/命令/网络）
  ├── 上下文管理器
  └── Claude API 调用

你在终端输入 claude，启动一个 Node.js 进程。这个进程里跑着完整的 Agent Loop，接收你的输入、调用 Claude API 推理、执行工具操作、把结果喂回模型、继续下一轮。

整个过程是同步阻塞的。模型在想的时候，你等着；工具在执行的时候，模型等着。一次只做一件事，但做得很深。

这种设计的好处是可预测，你能清楚地知道它在干什么、干到哪一步了。坏处是不能并行，一个会话同时只能处理一个任务。

2.2 Agent Loop 的三阶段

Claude Code 的 Agent Loop 分三个阶段，但不是严格线性的，更像是交织在一起的：

阶段一：收集上下文（Gather Context）

每一轮循环开始时，Agent 要搞清楚”我现在面对的是什么情况”。它会： - 读取你的输入 - 加载 CLAUDE.md（项目级配置文件，类似项目的”使用说明书”） - 加载已启用的 Skills 描述 - 加载 MCP 服务器的工具定义 - 回顾之前的对话历史

阶段二：执行动作（Take Action）

把上下文打包发给 Claude API，模型返回下一步操作。可能是： - 调用一个工具（读文件、执行命令、搜索代码等） - 直接给你一个回答 - 请求更多信息

如果是工具调用，执行完后把结果加入上下文，进入下一轮循环。

阶段三：验证结果（Verify Results）

模型拿到工具执行结果后，判断： - 任务完成了吗？ - 结果对不对？ - 需要继续操作吗？

如果没完成，回到阶段一继续。如果完成了，生成最终回复。

这三个阶段会循环很多轮。一个复杂任务可能要转几十圈，读代码、分析、改代码、跑测试、发现测试失败、再改、再跑，直到通过。

2.3 工具系统：三层设计

Claude Code 的工具分三层，我觉得这个设计很值得学习：

第一层：内置工具（Built-in Tools）

最核心的一组，开箱即用： - 文件操作：Read、Edit、Write - 搜索：Grep、Glob（文件名匹配） - 命令执行：Bash - 网络：WebFetch、WebSearch - 代码智能：跳转定义、查找引用

这些覆盖了 80% 的编程场景。

第二层：Skills（技能）

用户自定义的工作流模板。每个 Skill 是一个目录，里面有个 SKILL.md 文件描述这个技能能做什么、怎么做。

聪明的地方在于按需加载。系统提示词里只放 Skills 的名字和简短描述，不放完整内容。只有当模型判断需要用某个 Skill 时，才会去读取完整的 SKILL.md。

这样做的好处是省上下文。如果你装了 50 个 Skills，不可能把 50 个完整说明都塞进提示词里，上下文窗口会爆。

第三层：MCP 服务器（外部工具）

通过 MCP 协议连接外部服务。每个 MCP Server 暴露一组工具，Claude Code 把这些工具定义合并到系统提示词中。

但这里有个坑：每个 MCP Server 的工具定义都会占用上下文空间，即使你没在用它。当工具定义超过上下文窗口的 10% 时，Claude Code 会启动”工具搜索”机制，延迟加载工具，只在模型需要时才注入。

2.4 权限控制：安全是第一优先级

Claude Code 的权限系统设计得很细致，分几个层级：

权限模式： - default：每次敏感操作都问你（默认） - acceptEdits：自动接受文件编辑，但命令还是要确认 - plan：只读模式，只分析不动手 - bypassPermissions：完全跳过权限检查（危险，不建议）

权限规则： 可以精确到具体命令和文件路径。比如允许 npm run test 但禁止 rm -rf，允许读 .env 但禁止读 .env.production。

规则按 deny → ask → allow 的顺序匹配，第一个命中的生效。

沙盒机制： 在操作系统层面限制 Agent 能访问的文件和网络。权限控制的是”Agent 想做什么”，沙盒控制的是”Agent 实际能碰什么”。两层防护。

我从这套设计里学到的是：Agent 产品的安全不能只靠一层，要多层叠加。用户信任是 Agent 产品的生命线。

2.5 上下文管理

大模型有上下文窗口限制（Claude 目前是 200K token）。长时间工作后，对话历史会越来越长，最终撑满窗口。

Claude Code 的应对策略：

自动压缩（Compact）：当上下文接近上限时，自动总结之前的对话，用精简版替换原始内容。就像人的记忆，你不会记住每句话的原文，但记得关键信息。我在长时间使用中发现，压缩后偶尔会丢失一些细节，但整体体验还是流畅的。

子代理（Sub-agents）：把子任务派给独立的 Agent 会话处理。子代理有自己的上下文窗口，不会污染主会话。做完后只把结果返回。我用 Claude Code 的体验是，子代理在处理大文件重构时特别好用。

CLAUDE.md 项目记忆：项目级别的持久化配置。不管开多少次新会话，CLAUDE.md 里的内容都会被加载。适合存放项目约定、代码规范、常用命令等。

2.6 小结

Claude Code 的架构就像一个专注的手艺人，一次只做一件事，但做得精细。简单的架构带来了可预测性和可靠性，代价是不能并行、不能后台运行。

三、OpenAI Codex：云端并行的工厂模式

3.1 整体架构

Codex 的架构思路跟 Claude Code 完全不同。如果说 Claude Code 是一个手艺人，我觉得 Codex 更像一座工厂。

用户（ChatGPT / CLI / IDE）
  ↓
Codex 调度层
  ├── 任务 A → 云端沙盒 A（独立环境 + 代码仓库副本）
  ├── 任务 B → 云端沙盒 B（独立环境 + 代码仓库副本）
  ├── 任务 C → 云端沙盒 C（独立环境 + 代码仓库副本）
  └── ...

每个任务跑在独立的云端沙盒里，预加载了你的代码仓库。任务之间完全隔离，互不影响。

这意味着你可以同时给 Codex 5 个任务，它在 5 个独立环境里并行处理。一个任务失败了不影响其他的。

3.2 codex-1 模型

Codex 底层用的不是通用的 GPT 模型，而是专门为编程优化的 codex-1。

codex-1 基于 o3 模型，通过强化学习在真实编程任务上训练。它的特点是： - 生成的代码风格接近人类（不是那种一看就是 AI 写的） - 能理解项目的 PR 偏好和代码规范 - 遇到测试失败会自己迭代修改，直到测试通过

在 SWE-Bench Verified 基准测试上，codex-1 拿到了 75.6% 的通过率。

3.3 三种使用形态

Codex 不是一个产品，是一个产品家族：

Web 版（chatgpt.com/codex）：在 ChatGPT 里直接用。侧边栏点 Codex，连接 GitHub 仓库，给任务就行。最方便，适合日常使用。

CLI 版（命令行）：npm install -g @openai/codex，在终端里用。开源的（Apache-2.0），代码在 github.com/openai/codex。适合喜欢终端的开发者。

IDE 插件：VS Code、Cursor、Windsurf 都能装。在编辑器里直接跟 Codex 交互，还能把任务委托到云端执行。

三种形态共享同一个后端，选哪个看个人习惯。

3.4 GitHub 深度集成

Codex 跟 GitHub 的集成是它的一大卖点。

你可以在 GitHub 的 Issue 或 PR 里 @codex，它会自动创建一个云端任务去处理。比如有人提了一个 bug issue，你在评论里写 @codex 请修复这个 bug，它就会： 1. 克隆仓库到沙盒 2. 分析 issue 描述 3. 定位问题代码 4. 修复并跑测试 5. 创建一个 PR 给你审查

整个过程你不需要打开编辑器。

3.5 AGENTS.md 配置

跟 Claude Code 的 CLAUDE.md 类似，Codex 用 AGENTS.md 文件来配置项目级别的指令。

你可以在里面写： - 项目的目录结构说明 - 测试怎么跑 - 代码规范和偏好 - 哪些文件不要动

Codex 每次执行任务前都会读这个文件，就像新员工入职先看公司手册。

3.6 安全：沙盒隔离

每个任务跑在独立的云端沙盒里，默认不能访问互联网。这意味着： - 任务之间完全隔离 - Agent 不能把你的代码发到外面 - 不能下载恶意依赖

如果任务确实需要网络访问（比如安装依赖），可以手动开启。

3.7 小结

Codex 的架构像一座现代化工厂，标准化、可并行、有隔离。它把”编程 Agent”这件事做到了产品化的极致，但也因此局限在编程场景里。

四、OpenClaw：全场景的瑞士军刀架构

4.1 整体架构

OpenClaw 的架构是三者中最复杂的，因为它要解决的问题也最复杂。我自己在用 OpenClaw，所以这部分我会讲得更细一些。它不只是编程，而是”成为一个全能 AI 助手”。

多渠道入口（飞书/QQ/钉钉/Telegram/Discord/...）
  ↓
Gateway（网关守护进程）
  ├── 会话路由（哪个渠道来的消息，发到哪个会话）
  ├── 模型调度（选哪个模型处理这个请求）
  ├── 主会话 Agent Loop
  ├── 子代理会话池
  ├── Cron 定时任务引擎
  └── 心跳调度器

最大的架构特点是 Gateway + Client 分离。Gateway 是一个常驻后台的守护进程（daemon），7×24 小时运行。它负责接收各个渠道的消息、管理会话、调度模型。

这跟 Claude Code 和 Codex 都不一样。它们是”用的时候启动，用完就关”，OpenClaw 是”一直在线，随时待命”。

4.2 多渠道路由

OpenClaw 能同时连接十几个聊天渠道：飞书、QQ Bot、钉钉、企业微信、Telegram、Discord、Signal、WhatsApp……

Gateway 里有一个路由层，负责把不同渠道来的消息分发到正确的会话。你在飞书上跟它说的话和在 QQ 上说的话，可以是同一个会话（共享上下文），也可以是独立的会话。

这个设计让 Agent 真正变成了一个”跨平台助手”。我自己就是飞书和 QQ 两个渠道都在用，不管在哪个 App 里，都能找到它。

4.3 Skill 插件系统

OpenClaw 的工具扩展机制叫 Skill。跟 Claude Code 的 Skills 类似，但生态更开放。

每个 Skill 是一个独立目录，包含 SKILL.md（说明文件）和可能的脚本、配置。社区有一个 Skill 市场（clawhub.com），你可以一键安装别人做的 Skill。

我装了不少：天气查询、股票分析、YouTube 字幕提取、图片生成、PDF 处理……

Skill 的加载也是按需的，系统提示词里只放描述，用到时才读完整内容。

4.4 记忆系统：三层架构

这是 OpenClaw 最有意思的设计之一。

第一层：会话内记忆（短期） 就是当前对话的上下文。跟所有 Agent 一样，有窗口限制，会自动压缩。

第二层：日志记忆（中期） 每天的工作记录存在 memory/YYYY-MM-DD.md 文件里。Agent 每次启动会读最近两天的日志，了解近期发生了什么。

第三层：长期记忆（MEMORY.md） Agent 自己维护的”人生经验”。它会定期回顾日志，把重要的事情提炼到 MEMORY.md 里，比如用户的偏好、项目的状态、踩过的坑、学到的教训。

这三层记忆让 OpenClaw 有了”成长”的能力。用得越久，它越了解你。我用了一个多月，明显感觉它对我的工作习惯和偏好越来越熟悉。

还有一个 SOUL.md 文件，定义 Agent 的”性格”，包括说话风格、行为准则、价值观。你可以让它变得幽默、严肃、或者像某个特定的角色。

4.5 心跳机制

大部分 Agent 是被动的，你找它，它才响应。OpenClaw 不一样，它有心跳（Heartbeat）。

Gateway 会定期（默认每 30 分钟）给 Agent 发一个心跳信号。Agent 收到后，会按照 HEARTBEAT.md 里的清单做检查： - 有没有新的 GitHub 通知？ - 有没有即将到来的日程？ - 有没有需要跟进的任务？ - 记忆文件需不需要整理？

如果有需要处理的事情，它会主动给你发消息。如果没有，就安静地回一个”一切正常”。

这个设计让 Agent 从”被动工具”变成了”主动伙伴”。它不只是等你吩咐，还会主动关心你的事情。

4.6 Cron 定时任务

OpenClaw 内置了一个 Cron 引擎，可以设置定时任务。

比如： - 每天早上 9 点检查邮件 - 每周一和周四自动整理 AI 热文到 Notion - 每周三晚上 9 点做产品深度研究 - 5 分钟后提醒我喝水

每个 Cron 任务跑在独立的会话里（isolated session），不影响主会话。任务完成后可以通过各个渠道把结果推送给你。

4.7 模型无关

OpenClaw 不绑定任何一家模型厂商。你可以配置多个模型提供商，设置故障转移链：

主力模型挂了 → 自动切到备用模型 → 备用也挂了 → 切到第三个

我的配置是：Kimi K2.5 → Claude（代理）→ Gemini → Moonshot。哪个能用用哪个，用户无感知。

这在产品设计上是一个重要决策，不把鸡蛋放在一个篮子里。

4.8 子代理系统

跟 Claude Code 的子代理类似，OpenClaw 也能派子代理处理子任务。但它的子代理更灵活： - 可以指定不同的模型（主会话用 Opus，子代理用 Flash 省钱） - 可以设置超时时间 - 完成后自动把结果汇报回主会话 - 可以给子代理贴标签，方便管理

4.9 小结

OpenClaw 的架构像一个操作系统，有进程管理（会话）、有文件系统（记忆）、有定时器（Cron）、有网络（多渠道）、有插件（Skills）。它不是为某个场景优化的工具，而是一个通用的 Agent 运行平台。

六、MCP 协议：Agent 的”USB-C 接口”

三款产品都涉及到一个关键问题：Agent 怎么连接外部工具？

以前的做法是每个工具写一套定制代码。接 GitHub 写一套，接数据库写一套，接邮件写一套。换个 AI 平台，全部重写。

MCP（Model Context Protocol）改变了这个局面。Anthropic 在 2024 年 11 月发布了这个开放协议，到 2026 年已经成了事实标准。

MCP 的核心思路很简单：统一 Agent 连接工具的接口。就像 USB-C 统一了充电线，以前手机、电脑、耳机各用各的线，现在一根线搞定。

MCP 有三个角色： - Client（客户端）：Agent 这边。Claude Code、OpenClaw 都是 Client。 - Server（服务端）：工具那边。每个 MCP Server 封装一个或一组工具。 - Protocol（协议）：两边怎么对话的规矩。基于 JSON-RPC 2.0，格式统一。

写一个 MCP Server，所有支持 MCP 的 Agent 都能用。不用为每个 AI 平台单独适配。

Claude Code 和 OpenClaw 都原生支持 MCP。Codex 目前主要通过内置工具和 AGENTS.md 配置来扩展能力，MCP 支持在逐步完善中。

七、写在最后

三款产品，三种架构，三种设计哲学：

  • Claude Code 选择了极致专注，做好终端编程这一件事
  • Codex 选择了平台生态，融入 ChatGPT，用云端算力换并行能力
  • OpenClaw 选择了开放自由，让你自己定义 Agent 应该是什么样的

没有哪个架构是”最好的”。选择取决于你要解决什么问题、面向什么用户、愿意做什么取舍。
