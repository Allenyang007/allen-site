从大模型到 Agent：AI 应用的完整技术栈

写给所有人的 AI 技术栈全景指南——从能聊天到能干活，从单兵到团队

Allen2025-02-19约 10000 字阅读约 25 分钟AIAgentMCP

引言

2024 年以来，AI 领域最火的词从"大模型"变成了"Agent"。几乎每周都有新的 Agent 产品发布：Anthropic 的 Claude Code 能自主编写整个项目，Manus 号称能替你完成任何任务，OpenAI 推出的 Operator 可以替你操作浏览器购物订餐，各家 AI 助手纷纷从"能聊天"升级为"能干活"。

但这个领域的概念多到让人眼花缭乱：大模型、Token、Context Window、RAG、向量数据库、Vector Embedding、Agent、Function Calling、MCP、Skills、多 Agent 协作……每一个词背后都是一套技术体系，彼此关联、层层递进。你可能已经在日常工作中用上了 ChatGPT 或 Claude，但你真的理解它是怎么运作的吗？为什么它有时候会信誓旦旦地说错话？为什么 Agent 可以帮你自动执行任务？为什么最近 MCP 这么火？

这篇文章试图用一条清晰的线索，把整个 AI 应用技术栈从底层到顶层串起来。我会用通俗比喻解释每个概念，但也不回避技术原理——目标是让你读完后，既知道"是什么"，也理解"为什么"，还知道"用在哪"。

整个技术栈的逻辑递进关系如下：

大模型 LLM → RAG 检索增强 → Agent 智能体 → MCP 协议 → Skills 技能
大脑 → 图书馆 → 手和脚 → USB 接口 → App Store

大模型提供智能；RAG 弥补大模型的知识缺口；Agent 把大模型从"聊天者"升级为"执行者"；MCP 统一了 Agent 连接外部世界的接口标准；Skills 把能力封装成可即插即用的模块。这五层构成了当今 AI 应用的完整技术栈。

让我们从最底层的"大脑"开始。

一、大模型基础：AI 的「大脑」

在理解所有上层应用之前，必须先理解它们的基座——大语言模型（Large Language Model，LLM）。2022 年 11 月 ChatGPT 横空出世，让全世界人第一次真正体验到了 AI 的智能。但这只是冰山一角——在它背后，是几十年 AI 研究积累、数万亿个参数、数千张 GPU 日夜运转的庞大工程。

把大模型想象成一个读了全世界所有书的超级学霸。他读过维基百科的每篇文章、GitHub 上的每行代码、Reddit 上的每条讨论、知乎上的每个回答。他的知识面广到不可思议——上知天文下知地理，能写诗也能写代码，能分析股市也能帮你写求职信。但他有一个关键特点：他不会上网搜索，只能凭记忆回答你的问题。如果记忆不清晰，他甚至会自信满满地"编"一个听起来合理的答案——这就是"幻觉"（Hallucination）现象的由来，也是我们接下来需要引入 RAG 的根本原因。

Token 与上下文窗口：AI 的"阅读单位"

大模型不是逐字阅读文本的。它把文本切成一个个小块——Token（词元），这是 AI 的基本"阅读单位"。分词（Tokenization）过程由专门的分词器完成，规则因语言而异：

  • 中文：一个汉字通常对应 1-2 个 Token（有时多个字合为一个 Token）
  • 英文：一个常见词通常是 1 个 Token，罕见词可能被拆分成多个 Token
  • 代码：语法关键字通常是独立 Token，缩进和换行也是

举例："Hello, World!" 大约是 4 个 Token；"你好，世界！"大约是 6 个 Token。"你好世界"约 4 个 Token。模型的计费、速度、处理能力，都以 Token 为单位衡量。你在用 Claude API 时看到的价格，本质上是在为 Token 计费。

上下文窗口（Context Window）决定了模型一次能"看到"多少内容——类似人类的工作记忆容量。想象一张桌子上能同时摆多少纸张，这就是上下文窗口的比喻。

  • GPT-3.5-turbo：4K Token（约 3,000 词）
  • GPT-4 Turbo：128K Token（约 10 万字，相当于一本书）
  • Claude 3：最高 200K Token
  • Gemini 1.5 Pro：最高 1M Token（约 75 万字）
为什么上下文窗口很重要？

上下文窗口决定了 AI 能同时处理的信息量。窗口太小，你没法把整个文档给它看；窗口大了，它才能理解长对话的前后关系。但代价是：窗口越大，计算量越大（是平方级增长），推理越慢，单次调用成本越高。这是一个核心的工程权衡，也是 RAG 存在意义的重要背景——即使上下文窗口再大，也没法把所有文档都塞进去。

当你向 AI 提问时，模型做的事叫推理（Inference）。它不是从数据库里查答案，而是根据输入，逐词（准确说是逐 Token）预测最可能的下一个词，然后把这个词加入输出，再预测下下个词，如此循环，直到生成完整回答。

这个过程本质上是条件概率计算：P(下一个词 | 所有已知上下文)。"今天天气真" → "好" 的概率比 "今天天气真" → "差" 的概率高（大多数情况下）。模型会从这个概率分布中采样，这也是为什么每次问同一个问题，AI 给出的答案可能略有不同——它在从概率分布中采样，而不是确定性地输出。

Transformer 架构：为什么是"注意力"？

2017 年，Google Brain 发表了划时代的论文《Attention Is All You Need》，提出了 Transformer 架构，由此开创了现代大语言模型的新纪元。所有你今天用过的 LLM——GPT 系列、Claude 系列、Gemini、Llama——都是 Transformer 的变体。

Transformer 的核心创新是自注意力机制（Self-Attention Mechanism）。在理解它之前，先看看之前的主流方案——循环神经网络（RNN）的局限：RNN 像一个人从头到尾顺序读一本书，当他读到第 500 页时，第 1 页的内容已经淡忘大半。它处理长距离依赖关系的能力很弱。

自注意力机制的革命性在于：它让模型在处理每个词的时候，能同时"关注"到输入序列中的所有其他词，并动态计算每对词之间的相关性权重。

举个具体例子："法官宣判了被告，因为他对证据有充分的了解。"——这里"他"指的是"法官"还是"被告"？人类可以通过理解语境来判断，但 RNN 在处理"他"时，已经忘记了远处的"法官"和"被告"的语义关系。而自注意力机制能让"他"这个词同时关注句子中所有其他词，计算出"他"和"法官"的关联权重更高，从而正确理解指代关系。

从工程角度，自注意力还有一个关键优势：可以充分并行化。RNN 必须串行处理（每个时间步依赖上一步的结果），而 Transformer 的所有 Token 可以同时并行计算，极大地提升了训练效率，使得训练数千亿参数的模型成为可能。

Transformer 的另一个重要组件是位置编码（Positional Encoding）。自注意力机制天然不关心词序（"猫吃鱼"和"鱼吃猫"的词对权重是一样的），位置编码通过在每个 Token 的表示中加入位置信息，让模型能区分词序，从而理解语法结构。

训练三部曲：从白纸到专家

一个大模型从"什么都不懂"到"无所不能"，需要经历三个关键阶段。了解这三步，你就能理解为什么模型有时候聪明、有时候"蠢"，以及为什么 Claude 和 GPT 的"性格"会有差异。

第一步：预训练（Pre-training）

预训练是最耗资源的阶段。你需要收集尽可能多的文本数据——Common Crawl 爬取的网页、Wikipedia、GitHub 代码、书籍、科学论文……合计数万亿 Token。然后训练模型做一件事：预测被遮盖的词（或者预测下一个词）。

这个任务看起来简单，但要做好它，模型必须学会大量的知识和能力：语法、事实、常识、推理逻辑、代码语法……全部都以隐式的方式编码在模型的数十亿参数中。这一步让模型获得了广泛的世界知识，但它还只是一个"续写机器"——给它一段话，它会接着写，但不一定会回答你的问题。

预训练的成本令人咋舌：GPT-4 的训练花费估计超过 1 亿美元，使用了数千张 A100 GPU，耗时数月。这也是为什么只有少数大公司能做这件事。

第二步：指令微调（Supervised Fine-Tuning，SFT）

预训练后的模型会"续写"，但不会"回答"。你输入"法国的首都是哪里？"，它可能会续写出"法国的首都是哪里？这是一个常见的地理问题，很多人在初中地理课上学过……"而不是简洁地回答"巴黎"。

SFT 通过精心标注的"问题-回答"数据对来解决这个问题。数据格式通常是：

sft_data_format.json
{ "instruction": "请用中文解释什么是量子纠缠", "input": "", "output": "量子纠缠是量子力学中的一种现象……" }

有了几十万到几百万条这样的数据对，模型就学会了"当收到一个问题时，应该给出一个有用的回答"。这一步把模型从"语言模型"变成"对话助手"。但 SFT 数据的质量和多样性至关重要——数据里有多少种任务类型、回答有多精准，直接决定了助手的能力边界。

第三步：基于人类反馈的强化学习（RLHF）

SFT 之后，模型能回答问题了，但回答的质量参差不齐。有时候它会给出准确有用的答案，有时候会拐弯抹角、言之无物，甚至产生有害内容。RLHF（Reinforcement Learning from Human Feedback）的目标是让模型学会生成人类真正偏好的回答。

整个过程分三步：

  1. 收集偏好数据：让模型对同一个问题生成多个不同的回答，由人类标注员对这些回答进行排序（哪个更好）
  2. 训练奖励模型：用这些排序数据训练一个"奖励模型"（Reward Model），它能预测人类会对某个回答打多少分
  3. 强化学习优化：用强化学习（通常是 PPO 算法）来优化原始语言模型，使其生成能获得奖励模型高分的输出

RLHF 的效果显著：它让模型变得更有帮助（Helpful）、更无害（Harmless）、更诚实（Honest），这也是 Anthropic 所说的 HHH 原则。ChatGPT 的成功，很大程度上来自 RLHF 带来的良好对话体验。近年来，RLHF 的变体也在不断涌现，如 DPO（Direct Preference Optimization）等更高效的方法。

预训练
海量数据 → 知识 → SFT
标注数据 → 对话 → RLHF
人类反馈 → 质量

二、RAG：给 AI 配一个「图书馆」

理解了大模型的本质——一个凭记忆回答问题的"超级学霸"——你就能理解它的三个根本性局限：

  1. 知识有截止日期：训练数据有时间截止点。GPT-4 的训练数据截止于 2023 年初，它不知道此后发生的任何事情。
  2. 不含私有信息：你公司的内部文档、产品手册、客户数据——这些从未出现在互联网上，模型没见过，也无从知晓。
  3. 会产生幻觉：模型是在"预测下一个词"，而不是在"查询事实"。当它"不知道"时，它不会诚实地说"我不知道"，而是会生成听起来合理的内容——这就是幻觉（Hallucination）。

RAG（Retrieval-Augmented Generation，检索增强生成）就是为了解决这三个问题。它的核心思路简单而优雅：不要让模型单靠记忆回答，先让它"查资料"，再基于资料生成回答。

把 RAG 想象成开卷考试。一个聪明的学生参加开卷考试，不需要把教材背下来，只需要知道去哪找、怎么用。他拿到考题后，先翻书找到相关章节，然后基于书上的内容作答。RAG 让 AI 在回答之前先"翻书"——而不是全靠死记硬背。答案有依据，可以溯源，也不容易"编造"。

为什么需要 RAG：三个真实场景

场景一：企业知识库问答

一家公司想让 AI 回答员工的 HR 问题："我的年假还剩多少天？报销医疗费用需要什么材料？晋升机制是怎样的？" 这些内容在公司内部文档里，但模型训练数据里没有。直接问模型，它只能瞎猜或者给出通用回答。用 RAG，把公司的 HR 政策文档索引起来，员工提问时先检索相关段落，再让模型基于这些段落回答——答案精准、可溯源、不会出现幻觉。

场景二：实时信息查询

你问 AI："特斯拉今天的股价是多少？"模型不知道——它的训练数据几个月甚至一年前就截止了。但如果给 Agent 配备一个"搜索工具"，它可以实时搜索财经网站，把最新数据检索回来，再生成回答。这其实是 RAG 思路在工具调用层面的延伸。

场景三：长文档分析

你有一份 200 页的法律合同，想让 AI 回答"第三方责任条款里有哪些例外情况？"整个合同可能超过上下文窗口限制，就算塞进去也会因为信息密度过高而降低质量。RAG 先把合同切分成段落，检索最相关的几段，再让模型精确回答——效率更高，效果更好。

向量化与向量数据库：语义搜索的核心

RAG 的核心技术是向量化（Vector Embedding）。理解它之前，先思考一个问题：传统搜索（如 Elasticsearch）为什么不够用？

传统关键词搜索做的是词汇匹配——文档里包含你搜索的词语就匹配，不包含就不匹配。这有两个致命问题：

  • 同义词问题：搜"购买"找不到含"买入"的文档
  • 语义理解问题：搜"苹果很贵"，包含"苹果"和"贵"的文档都会匹配，但"苹果手机很贵"和"苹果水果很贵"的语义完全不同

想象一个巨大的多维坐标系。把每个概念映射到这个坐标系里的一个点。"小狗"在一个位置，"柯基""金毛""哈士奇"在它附近；"汽车"在很远的地方；"car"（英文）和"汽车"在相近的位置，因为它们语义相同。这个坐标系，就是语义空间。向量化（Embedding）就是把文本映射到语义空间的过程。

技术上，向量化由专门训练的 Embedding 模型完成（如 OpenAI 的 text-embedding-ada-002、OpenAI 的 text-embedding-3-large，或开源的 bge-m3）。它把任意一段文本转换成一个固定长度的浮点数数组——比如 1536 维。每个维度代表文本在某个语义特征上的"强度"。两段文本语义越相近，它们的向量在空间中越靠近（余弦相似度越高）。

embedding_example.py
import openai # 将文本转换为向量 response = openai.embeddings.create( model="text-embedding-ada-002", input="什么是机器学习？" ) vector = response.data[0].embedding # vector 是一个长度为 1536 的浮点数列表 # [-0.0067, 0.0123, -0.0089, ..., 0.0045] # 语义相近的文本，向量之间的余弦相似度接近 1.0

向量数据库（Vector Database）专门用于存储和高效检索这些向量。主流选择包括：

  • Pinecone：云原生向量数据库，运维成本低，适合快速上线
  • Milvus：开源，高性能，适合大规模生产部署
  • Qdrant：Rust 编写，内存效率高，支持过滤查询
  • pgvector：PostgreSQL 扩展，适合已有 PG 技术栈的团队
  • Chroma：轻量级，适合本地开发和小规模应用

向量数据库的核心算法是近似最近邻搜索（Approximate Nearest Neighbor，ANN）。在百万级向量中精确找到最相似的向量，计算量是 O(n)，太慢。ANN 算法通过构建特殊的索引结构（如 HNSW——分层小世界图算法），在轻微牺牲精度的前提下，把查询复杂度降到 O(log n)，做到毫秒级响应。

RAG 的完整工作流

文档 → 切分 Chunk → Embedding → 向量数据库
——离线索引——

用户提问 → 问题向量化 → 向量检索 top-k → 组合 Prompt → LLM 生成
——在线查询——
rag_pipeline.py
# RAG 完整流程伪代码 # === 离线阶段：建立索引 === def build_index(documents, vector_db): for doc in documents: # 将文档切分成段落（Chunk） chunks = split_into_chunks(doc, chunk_size=512, overlap=50) for chunk in chunks: # 将每个段落转为向量 vector = embed(chunk.text) # 存入向量数据库 vector_db.upsert(id=chunk.id, vector=vector, metadata={"text": chunk.text}) # === 在线阶段：查询检索 === def rag_query(question, vector_db, llm): # 将问题向量化 query_vec = embed(question) # 检索最相关的 5 个段落 results = vector_db.search(query_vec, top_k=5) context = "\n\n".join([r.text for r in results]) # 组合 Prompt 并生成回答 prompt = f"""请基于以下资料回答问题，如果资料中没有相关信息，请直接说明。 参考资料： {context} 用户问题：{question}""" return llm.generate(prompt)
实际工程中 RAG 的核心挑战

RAG 看起来简单，但实际落地时有很多细节：如何切分文档（Chunking Strategy）——切得太小，缺乏上下文；切得太大，引入无关信息。如何处理表格和图片——纯文本向量化效果好，但结构化数据和图像需要特殊处理。如何处理多跳问题——"A 的上司的老家在哪里？"需要多次检索才能回答。工程实践中，切分策略往往决定了 RAG 系统 70% 的效果。

三、Agent：AI 有了「手」和「脚」

有了大模型提供"大脑"、RAG 提供"图书馆"，AI 已经能回答各种问题了。但它还只是在"说"——无法真正"做"任何事情。用户想让 AI 帮他发一封邮件、查一下数据库、提交一个代码 PR，大模型本身做不到，因为它被限制在一个"文本进、文本出"的黑盒里。

Agent（智能体）的出现，改变了这一切。

如果大模型是大脑，RAG 是图书馆，那么 Agent 就是让 AI 拥有了"手"和"脚"。一个普通聊天机器人只能对话——你问天气，它说"我无法查看实时天气"。一个 Agent 不同：你问天气，它自己调用天气 API 查询后告诉你结果；你让它订机票，它自己打开浏览器搜索航班、比价、选座、填写支付信息。Agent 不只是"说"，它还能"做"——而且能自主地做复杂的多步骤任务。

感知-规划-执行-反馈：Agent 的核心循环

Agent 的运行机制是一个持续的循环，通常被称为 ReAct（Reasoning + Acting）循环，或者简称为 Agent 循环：

👁 感知 Perceive → 🎯 规划 Plan → 🔧 执行 Act → 📊 反馈 Reflect → (循环)

感知（Perceive）：Agent 接收来自外界的信息——可能是用户的指令、环境的状态（比如当前浏览的网页内容），或者上一步工具调用的返回结果。这些信息都会进入 Agent 的上下文窗口。

规划（Plan）：这是大模型发挥核心作用的地方。模型分析当前任务，思考"为了完成目标，下一步应该做什么"。对于复杂任务，它会分解成子任务，制定执行策略。这个思考过程有时候是隐式的（直接输出行动），有时候是显式的（先输出思考过程 Chain of Thought，再决定行动）。

执行（Act）：模型决定调用某个工具，并输出结构化的调用请求。外部系统（Agent 框架）接收这个请求，实际执行工具调用，并把结果返回给模型。执行层涵盖一切"与外部世界交互"的操作：调用 API、运行代码、操作浏览器、读写文件等。

反馈（Reflect）：模型收到工具执行的结果，评估：任务是否完成了？结果是否符合预期？是否需要调整策略？如果任务未完成，继续下一轮循环；如果完成了，生成最终回答。

这个循环的关键在于：模型自主决定需要哪些工具、以什么顺序使用它们、何时停止。这和传统程序截然不同——传统程序的执行路径是人类写死的，而 Agent 的路径是模型动态规划的。这赋予了 Agent 处理开放性任务的能力，但也引入了不确定性和安全风险（后面会讲）。

工具调用（Function Calling）：Agent 的"双手"

工具调用（Function Calling，也叫 Tool Use）是 Agent 能力的核心实现机制。2023 年 6 月，OpenAI 在 GPT-3.5 和 GPT-4 API 中引入 Function Calling 功能，Agent 应用从此进入爆发式发展阶段。

工作原理如下：开发者在 API 请求中声明一组"工具"（Tools），每个工具有名称、描述和参数定义。模型在思考过程中，如果判断需要使用某个工具，就会在输出中包含一个结构化的工具调用请求（而不是普通文本）。外部系统检测到调用请求后，执行相应操作，把结果作为新消息返回给模型，模型继续推理。

function_calling_complete.py
import openai, json # 定义工具（告诉模型有哪些工具可用） tools = [{ "type": "function", "function": { "name": "get_weather", "description": "查询指定城市的实时天气", "parameters": { "type": "object", "properties": { "city": {"type": "string", "description": "城市名称"}, "date": {"type": "string", "description": "日期，默认 today"} }, "required": ["city"] } } }] # 第一轮：发送用户消息 messages = [{"role": "user", "content": "北京今天天气怎么样？"}] response = openai.chat.completions.create( model="gpt-4", messages=messages, tools=tools ) # 模型决定调用 get_weather 工具 tool_call = response.choices[0].message.tool_calls[0] # tool_call.function.name = "get_weather" # tool_call.function.arguments = '{"city": "北京", "date": "today"}' # 执行工具（由外部系统处理） weather_data = call_weather_api("北京", "today") # weather_data = {"temp": "22°C", "condition": "晴", "humidity": "45%"} # 第二轮：把工具结果返回给模型 messages.append(response.choices[0].message) messages.append({ "role": "tool", "tool_call_id": tool_call.id, "content": json.dumps(weather_data) }) final = openai.chat.completions.create(model="gpt-4", messages=messages) # 输出："北京今天天气晴朗，气温 22°C，湿度 45%，是个好天气！"

这个机制的设计哲学值得细品：模型只负责"决策"（调什么工具、传什么参数），而不负责"执行"（实际的 API 调用、权限验证、错误处理都在外部系统）。这种分离带来了巨大的灵活性和安全性——你可以在外部系统层面加入权限控制、审计日志、错误重试等机制，而不需要改变模型本身。

四、MCP 协议：AI 世界的「USB 接口」

Function Calling 让 AI 能用工具，解决了"能不能用"的问题。但随着 AI 应用的爆炸式增长，一个新问题出现了：每个 AI 应用都要为每个工具单独写对接代码，重复劳动极其严重。

Cursor 连接 GitHub 要写一套代码；Claude Desktop 连接 GitHub 又要写一套。Cursor 连接 Slack 要写，Claude Desktop 连接 Slack 又要写。这就像 USB 发明之前的世界——打印机有自己的接口，键盘有自己的接口，鼠标又有自己的接口，换一台电脑就要换一套线。

MCP（Model Context Protocol，模型上下文协议）就是 AI 世界的 USB 接口。它由 Anthropic 于 2024 年 11 月发布并开源，是一套开放的标准通信协议，旨在统一 AI 应用与外部工具、数据源之间的连接方式。

MCP 解决什么问题

在 USB 出现之前，每个外设都有自己独特的接口——打印机一种、键盘一种、鼠标又一种。USB 统一了这一切：任何符合 USB 标准的外设，都能插到任何支持 USB 的电脑上。MCP 做的是同样的事：在 MCP 之前，每个 AI 应用对接每个工具都需要单独开发；有了 MCP，任何符合 MCP 标准的工具（MCP Server），都能被任何支持 MCP 的 AI 应用（MCP Host）使用。

量化来看：假设有 N 个 AI 应用和 M 个工具。没有 MCP，需要写 N×M 套对接代码。有了 MCP，只需要 N 个 MCP Client 实现 + M 个 MCP Server 实现 = N+M 套代码。当 N=100，M=500 时，差距是 50,000 vs 600。这是开放标准协议的网络效应。

MCP 的架构层次

MCP 采用了经典的 Client-Server 架构，分四个层次：

MCP Host
AI 应用（Claude Desktop、Cursor、OpenClaw）
↓ 通过 MCP 协议通信
MCP Client
协议客户端，嵌入在 Host 中，管理连接
↓
MCP Server
工具提供方，封装具体能力为 MCP 标准接口
↓
Tools / Resources / Prompts
实际的外部资源——GitHub、数据库、文件系统
  • MCP Host：用户直接交互的 AI 应用。它是整个流程的起点，负责收集用户指令，决定调用哪些 MCP Server。
  • MCP Client：嵌入在 Host 内部的协议层，负责与 MCP Server 建立连接（通过 stdio 或 HTTP SSE），发送工具调用请求，接收并解析响应。
  • MCP Server：第三方开发的工具提供方。它把自己的能力（GitHub API、文件操作、数据库查询等）封装成符合 MCP 标准的接口，对外暴露。每个 Server 可以运行在本地进程或远程服务上。
  • Tools / Resources / Prompts：MCP Server 对外暴露的三种能力类型（详见下一节）。

MCP 使用 JSON-RPC 2.0 作为通信协议，支持两种传输方式：本地进程间的 stdio 通信（低延迟，适合本地工具），以及 HTTP Server-Sent Events（SSE，适合远程工具）。

MCP 的三大核心概念

Tools（工具）——可被 AI 主动调用的函数或操作。这是 MCP 中最核心、最常用的概念。

每个 Tool 有：名称（name）、描述（description，让模型知道何时调用它）、输入参数模式（JSON Schema 格式）、输出格式。模型根据描述判断何时调用，Framework 负责实际执行并把结果返回。

示例 Tool：GitHub 的 "create_issue" 工具，接受 repo、title、body 参数，创建一个 GitHub Issue 并返回 Issue URL。

Resources（资源）——AI 可以读取的数据源，是只读的。类比：Tools 是"手"（可以操作和改变世界），Resources 是"眼睛"（可以观察和获取信息）。

Resource 通过 URI 标识（如 file:///home/user/report.md 或 github://repo/main/src/app.py）。模型可以请求读取特定资源，获取上下文信息后再决定如何行动。Resource 是只读的，这限制了 AI 对数据源造成意外修改的风险。

Prompts（提示模板）——预定义的交互流程模板。它把"如何完成某类任务"的流程封装成可复用的模板，降低了用户的使用门槛。

例如，一个"代码审查"Prompt 模板会引导 AI：先读取代码文件（Resource），然后按照特定的审查维度（安全性、性能、可维护性）进行分析，最后调用创建 Review Comment 的工具（Tool）。用户只需要触发这个模板，不需要自己组织复杂的交互流程。

MCP 生态现状

截至 2025 年初，MCP 生态已经非常活跃。官方提供了 GitHub、文件系统、SQLite、Slack 等参考实现；社区已发布数百个 MCP Server，覆盖 Notion、Jira、Google Drive、Linear、AWS 等主流服务。支持 MCP 的 Host 包括 Claude Desktop、Cursor、VS Code（通过插件）、OpenClaw 等。MCP 的开放性使其正在成为 AI 工具集成的事实标准。

五、Skills 技能系统：AI 的「App Store」

MCP 解决了工具连接的标准化问题，但仍然是一个开发者面向的概念——你需要理解协议细节，知道如何启动 Server，知道哪些工具该在什么时候用。对于普通用户来说，这还是太复杂了。

Skills（技能系统）是在 MCP 之上的更高层抽象，目标是让 AI 的能力扩展变得像"安装 App"一样简单。

想象一下 iPhone 的 App Store。你不需要知道某个 App 是怎么实现的，不需要配置数据库连接或 API 密钥——你只需要点击"安装"，打开 App 就能用。Skills 对 Agent 能力做同样的事：一个 Skill 是一个完整的"能力包"，把工具调用、提示词模板、配置脚本、使用说明全部打包在一起，用户安装后即可使用，无需关心底层细节。

以 OpenClaw 的技能系统为例，一个"股票分析"Skill 可能包含：

  • 工具：连接财经 API 的 MCP Server（获取股价、财报数据、K 线数据）
  • 脚本：自动安装依赖、配置 API Key 的安装脚本
  • 指令：告诉 Agent "当用户询问股票时，应该先获取基本面数据，再分析技术指标，最后综合给出建议"
  • Prompt 模板：标准化的分析报告格式

安装完"股票分析" Skill 后，你只需对 Agent 说"帮我分析茅台最近的走势"，它就会自动执行完整的分析流程，输出结构化的报告。你不需要知道它调了哪些 API，也不需要配置任何参数。

维度MCP（底层协议）Skills（能力封装）
面向对象开发者终端用户
关注点如何连接工具能完成什么任务

类比USB 接口规范App Store 里的应用
使用门槛需要理解协议细节一键安装即用
包含内容工具接口定义指令 + 脚本 + 工具 + 资源 + 使用说明

Skills 的理念是：把能力封装到极致，让 Agent 的扩展和使用像 App Store 一样流畅。未来的 AI 助手生态，很可能就是"一个通用 Agent 基座 + 用户按需安装的 Skills 集合"，就像今天的智能手机——手机本身是通用硬件，真正的价值来自 App Store 里丰富的应用生态。

六、Agent 记忆系统：让 AI「记住」你

到目前为止，我们讨论的 Agent 都是"无记忆"的——每次对话开始时，它都是一张白纸，不记得上次说了什么，不知道你是谁，不了解你的偏好。这显然不是一个理想的助手。

真正有用的 AI 助手需要记忆系统——不只是记住当前对话，还要记住你是谁、你过去的需求、你的偏好和习惯。这就是 Agent 记忆系统要解决的问题。

想象一个新员工的成长过程。刚入职时，他每天都要重新问"这个流程是什么？那个数据在哪里？"——因为他没有任何记忆积累。工作几个月后，他记住了公司的流程、同事的特点、常用数据的位置，工作效率大幅提升。Agent 的记忆系统，就是要让 AI 完成这种从"新人"到"老手"的蜕变。

短期记忆 vs 长期记忆

短期记忆（Working Memory / In-Context Memory）——就是模型当前上下文窗口中的内容。Agent 记得你这轮对话中说过的每句话、调用过的每个工具、得到的每个结果。这些信息随着对话进行而积累，但当对话结束、上下文被清空时，一切都消失了。就像人挂了电话就忘了刚才说的号码。受限于上下文窗口大小（几万 Token），无法保存过多信息。

长期记忆（Long-term Memory / External Memory）——通过外部存储持久化保存的信息。Agent 可以把重要信息写入文件、数据库或向量存储，下次启动时再读取出来。这样即使关闭会话，Agent 仍然"记得"你的工作习惯、项目进展、个人偏好。

长期记忆的技术实现通常有三种方案，各有优劣：

  • 文件存储：最简单直接，把记忆写成 Markdown 或 JSON 文件。优点：可读可编辑，方便调试；缺点：检索效率低，大量记忆时无法快速找到相关内容。适合小规模、高可读性要求的场景。
  • 向量存储：把记忆内容向量化后存入向量数据库，检索时通过语义相似度匹配。优点：语义检索能力强，能找到相关但不完全匹配的记忆；缺点：需要额外的 Embedding 成本，实现复杂度高。适合大量非结构化记忆（如长期对话历史、用户偏好描述）。
  • 结构化数据库：用传统关系型数据库（MySQL、PostgreSQL）存储结构化记忆（用户偏好表、任务历史表等）。优点：查询精确，事务支持好；缺点：需要预定义 Schema，灵活性差。适合结构清晰、需要精确查询的场景（如"用户上次搜索的航班日期"）。

实际产品中，往往是多种方案的组合。以 OpenClaw 为例，它使用文件系统存储日常记忆（memory/YYYY-MM-DD.md），同时维护一个长期记忆文件（MEMORY.md）保存精炼后的关键信息——类似人类把日记里的重要发现整理到"知识库"里。Agent 在每次对话开始时读取这些文件，获取长期上下文。

memory_system_design.md
# OpenClaw 记忆系统架构（简化） ## 短期记忆（当前对话上下文） - 存储位置：模型上下文窗口（in-memory） - 生命周期：当前会话 - 容量：受上下文窗口限制（~200K Token） ## 长期记忆（文件系统） workspace/ ├── MEMORY.md # 精华长期记忆（类似人的长期记忆） ├── memory/ │ ├── 2025-01-15.md # 每日日记（原始事件记录） │ └── 2025-01-16.md ├── USER.md # 用户画像（偏好、背景） └── PROJECT.md # 当前项目状态 # 每次对话开始时，Agent 读取这些文件， # 把关键信息注入上下文（短期记忆 + 长期记忆结合）

七、多 Agent 协作：分工带来超能力

一个 Agent 再强大，也有它的局限：单一的上下文窗口限制了它能同时处理的信息量；单一的知识和能力无法兼顾所有领域；单线程的执行方式无法并行处理多个子任务。

就像一个再优秀的员工，也无法同时完成设计、开发、测试、运营的所有工作——我们需要团队协作。多 Agent 协作（Multi-Agent Collaboration）正是把这种团队协作模式引入 AI 系统。

分工模式与编排调度

多 Agent 系统中，最常见的分工模式有两种：

流水线模式（Pipeline）：任务按顺序在多个 Agent 之间流转，每个 Agent 负责处理某个特定环节。类比工厂流水线：原材料进入，经过多道工序，成品出来。

例如一个"内容生产"流水线：研究 Agent（搜集资料）→ 写作 Agent（撰写初稿）→ 编辑 Agent（修改润色）→ 发布 Agent（格式化并发布）。每个 Agent 专注于自己最擅长的环节，整体质量高于单一 Agent 全部完成。

编排模式（Orchestration）：一个"主控 Agent"（Orchestrator）负责分解任务、分配给专业 Sub-Agent 并协调结果。类比项目经理带领团队：PM 制定计划，把不同的任务分配给设计师、开发、测试，最后汇总验收。

编排器 Agent
任务分解、分配、协调
↓
代码编写 Agent 测试验证 Agent 文档撰写 Agent

多 Agent 协作的核心挑战在于信息共享与一致性。Agent 之间需要共享上下文，但每个 Agent 的上下文窗口是独立的，不能简单地把所有信息传给每个 Agent（会超出窗口限制）。如何设计高效的信息传递机制，确保每个 Agent 拥有完成自己任务所必需的上下文，同时不被不必要的信息淹没，是多 Agent 系统设计的核心难题。

另一个挑战是错误传播：如果某个 Agent 产生了错误结果，后续 Agent 可能在错误基础上继续执行，导致最终结果完全偏离预期。因此，多 Agent 系统通常需要设计检查点（Checkpoint）和验证机制。

八、产品图鉴：谁在做 Agent？

从概念到产品，AI Agent 赛道已经涌现了一批优秀的工具。这里精选六款最具代表性的产品，从不同维度进行深度分析。

Claude Code（Anthropic）

Claude Code 是 Anthropic 官方出品的 AI 编程助手，运行在命令行终端里。它不同于一般的"代码补全工具"——它是一个完整的 Agent：能读取整个代码库、理解架构设计、自主规划并执行多步骤的开发任务（比如"帮我重构这个模块的错误处理逻辑"或"帮我写完这个功能并补全单元测试"）。

Claude Code 的核心优势是深度代码理解——基于 Claude 3 系列模型，它对代码结构、语言特性、工程惯例有深刻理解。它原生支持 MCP 协议，可以连接 GitHub、数据库等外部工具，形成完整的开发工作流。最适合技术团队在真实项目中使用，而不是单纯"生成代码片段"的场景。

Cursor

Cursor 是 AI 原生代码编辑器，基于 VS Code 深度定制。它的核心差异化在于把 AI 深度嵌入编辑器体验，而不是作为外挂插件。Tab 键驱动的智能补全、多文件同时编辑的 Composer 模式、自然语言驱动的代码修改——这些功能让 AI 辅助开发的体验极为流畅，学习成本几乎为零。

Cursor 支持多模型后端（GPT-4、Claude 等），同时也支持 MCP 协议，可以扩展连接外部工具。它最大的壁垒是用户习惯——一旦习惯了 Cursor 的工作流，很难回到普通编辑器。适合所有开发者，尤其是希望最小化学习成本的用户。

Dify

Dify 是开源的 LLM 应用开发平台，定位是让技术和非技术人员都能构建 AI 应用。它的核心特点是可视化工作流编排——通过拖拽节点来构建 AI 工作流，无需写代码。内置 RAG 引擎，支持多种文档格式（PDF、Word、网页等）直接上传建库。Agent 构建器让你通过配置界面创建具有工具调用能力的 AI 助手。

Dify 完全开源（GitHub Star 超过 60K），可以私有化部署，数据完全在自己手中。它支持对接各种 LLM 后端（OpenAI、Claude、Gemini、本地的 Ollama 等），灵活度极高。最适合产品经理、运营团队，或者需要快速构建 AI 应用的中小企业。

n8n

n8n 是开源的工作流自动化平台，在 AI 时代获得了"第二春"。它的核心能力是连接万物——内置 400+ 服务集成（Slack、Gmail、Notion、数据库、HTTP 等），通过可视化拖拽构建自动化工作流。它的 AI Agent 节点让 AI 成为工作流中的决策者：不只是机械执行固定流程，而是能根据情况动态调整。

n8n 支持自托管（完全掌控数据），也有云服务版本。它的典型用例是：监控邮件 → AI 判断优先级 → 自动回复或转到人工 → 记录到 CRM；或者定期从多个数据源汇总数据 → AI 分析 → 自动发送报告。适合运营、产品和技术团队的流程自动化。

Manus

Manus 是 2025 年初横空出世的通用型 AI Agent，一经发布就刷屏科技圈。它的定位是真正能自主完成任务的 Agent：给它一个目标，它自己规划路径、搜集信息、写代码执行、处理结果，最终交付完整产出。

Manus 能浏览网页、搜索信息、写并运行代码、读写文件、处理复杂的多步骤任务。它的技术架构基于多 Agent 协作——不同的子任务由专门的 Agent 处理，主控 Agent 负责协调。最适合需要 AI 完成"完整工作任务"而不只是"辅助"的场景——比如"帮我做一份竞品分析报告"或"帮我开发一个简单的数据可视化 Dashboard"。

OpenClaw

OpenClaw 是一个个人 AI Agent 运行时框架，走的是完全不同的路线——把 Agent 运行在你自己的服务器上，数据完全由你掌控。它支持多渠道接入（微信、QQ、飞书、Discord、Telegram），你可以通过任何熟悉的通讯工具和你的 Agent 交互。

OpenClaw 的技能系统（Skills）让能力扩展变得简单，原生支持 MCP 协议，可连接各种外部工具。它的记忆系统支持长期学习——Agent 会记住你的偏好、工作习惯、项目进展，越用越懂你。完全开源，可自由定制，适合技术爱好者和极客——如果你想拥有一个真正属于自己的、完全可控的 AI 助手，OpenClaw 是目前最佳选择之一。

九、未来展望：从工具到伙伴

在结束这篇文章之前，让我们抬头看看更远的地方。AI Agent 的技术栈正在快速成熟，但我们可能还只是在这个时代的最开始。

每个人的数字分身

想象五年后：你有一个 AI Agent 完全了解你的工作习惯——你喜欢的写作风格、你常用的工具、你的工作节奏、你的知识体系。它帮你筛选和处理邮件（但需要你签名的才发出），整理日程（主动建议推迟不重要的会议），撰写报告初稿（基于你历史的写作风格和你的最新数据），管理项目进展（定期汇报并预警风险）。

这不是一个聊天机器人，而是你的"数字分身"——一个 7×24 小时不休息的"你"。它不只是工具，更像是你能力的延伸。这样的 Agent 需要深度的个性化——它需要知道你是谁、你的价值观是什么、什么是你会批准的、什么是你会否决的。这就是 Agent 记忆系统和个性化能力的终极应用场景。

从工具到伙伴

当前的 AI 还是"工具"：你下指令，它执行；你不说话，它沉默。未来的 Agent 将更接近"伙伴"——它有主动性，能在你没想到的时候提供帮助；它有持续学习能力，越用越懂你；它有某种程度的"关心"——不只是完成任务，而是关注你的状态和需求。

"我注意到你这周三天都有超过五个会议，且明天还有一个大的 Deadline。要不要我帮你把最后两个非紧急会议推迟到下周？同时我已经根据你的历史报告草拟了明天 Deadline 的初稿，你方便时可以过目。"这种主动的、有同理心的、记得上下文的协作，是 Agent 进化的方向。

安全、隐私与可控性：不可回避的挑战

Agent 越强大，风险越大。当 AI 能自主操作你的电脑、访问你的数据、代表你发邮件和消息时，以下问题变得极为重要：

  • 权限边界：Agent 能做什么、不能做什么？敏感操作（删除数据、发送消息、支付）是否需要人类确认？
  • 数据安全：你的对话记录、个人信息、工作文件——存储在哪里？谁能访问？会不会被用于训练？
  • 可审计性：Agent 做了什么决策、为什么这么做——能不能事后查看和审计？
  • 对齐问题：如果 Agent 的目标和你的真实意图产生偏差，会发生什么？如何确保 Agent 的行为符合你的价值观？

这些问题的答案，将决定 AI Agent 能走多远、走多快。技术发展的速度远快于治理框架的完善速度，这是我们这个时代面临的核心挑战之一。OpenClaw 选择本地部署、完全开源的路线，某种程度上就是对"数据主权"这个问题的一种工程回答——最安全的 Agent，是运行在你自己服务器上的 Agent。

写在最后

从大模型到 Agent，我们正在经历 AI 应用的范式转变——从"能聊天"到"能干活"，从"工具"到"伙伴"。这个技术栈还在快速演进，今天的最佳实践可能明天就会被颠覆。但底层逻辑是清晰的：大模型提供智能，RAG 弥补知识缺口，Agent 赋予行动力，MCP 统一连接接口，Skills 降低使用门槛。理解了这条技术主线，你就能在这个快速变化的领域中保持方向感——不论明天又出现了什么新词、新工具、新产品。
