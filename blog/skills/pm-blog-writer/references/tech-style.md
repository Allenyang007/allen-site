# 写作风格指南 — 技术深度文章

## 定位

系统性的技术教程或深度分析，面向想深入理解某个技术领域的人。
参考已有文章：《从大模型到 Agent：AI 应用的深度技术指南》（15000字+目录侧栏）

## 特点

- **长篇系统**：5000-15000字，章节清晰
- **有深度**：不是入门科普，要讲清原理和工程细节
- **结构严谨**：大章节（h2）+ 子章节（h3），每个都有 id 用于目录锚点
- **代码和图表**：适当包含代码块、对比表、架构图（ASCII 或描述）
- **客观但有判断**：技术对比时给出自己的推荐和理由

## 结构

1. 引言：为什么写这篇，读完能获得什么
2. 核心章节 × 3-6 个：每个章节解决一个问题
3. 对比/总结：横向对比或实践建议
4. 参考资料：列出引用来源

## HTML 要求

- 必须有目录侧栏（.toc + .lay 双列布局）
- 支持代码高亮块（.cb + .cbd）
- 支持对比表格（.cpt）
- 支持提示框（.co-i / .co-t / .co-w）

## 去 AI 味（联动 humanizer skill）

技术文章也必须过 humanizer 审查。技术文章的 AI 味通常表现为：
- 过度使用"值得注意的是"、"需要强调的是"等连接词
- 每个技术概念后面都跟一段"为什么重要"的吹捧
- 对比表格里用"强大的"、"卓越的"等形容词
- 代码示例前后用"让我们来看看"、"正如你所见"
- 结构过于对称（每节都是：概念→原理→示例→总结）

技术文章可以更客观，但仍然需要作者的判断和偏好。"我更推荐 X，因为..."比"X 和 Y 各有优劣"有用得多。

## 传播与运营视角

技术长文也要有传播力，不能写成论文：

- **标题要抓人**：用"从X到Y"、"一篇讲透"、问句等结构，让非技术读者也想点开
- **开头3秒**：用一个具体场景或反直觉判断开场，不要上来就铺概念
- **埋钩子**：每个大章节结尾留一个悬念或过渡，让人想看下一节
- **金句**：技术文章也要有2-3句可截图传播的观点句，放在关键位置
- **结尾有余味**：不要机械总结，给一个面向未来的判断或一句有力量的收尾

## 元数据

- 标签: 从 [AI技术, Agent, 架构设计, 工具对比, 开发实践, 开源项目] 中选
- 阅读时间: 按 300字/分钟估算（技术内容读得慢）
