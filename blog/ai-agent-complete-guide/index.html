从大模型到 Agent：AI 应用的完整技术栈

写给所有人的 AI 技术栈全景指南——从能聊天到能干活，从单兵到团队

Allen2025-02-19约 10000 字阅读约 25 分钟AIAgentMCP

引言

2024 年以来，AI 领域最火的词从大模型变成了Agent。几乎每周都有新的 Agent 产品发布：Anthropic 的 Claude Code 能自主编写整个项目，Manus 号称能替你完成任何任务，各家 AI 助手纷纷从能聊天升级为能干活。

但概念多到让人眼花缭乱：大模型、Token、RAG、向量数据库、Agent、Function Calling、MCP、Skills、多 Agent 协作……每一个词背后都是一套技术体系，彼此关联、层层递进。

这篇文章试图用一条清晰的线索，把整个 AI 应用技术栈从底到顶串起来。用通俗比喻解释每个概念，但也不回避技术原理——目标是让你读完后，既知道是什么，也理解为什么。

大模型 LLM → RAG 检索增强 → Agent 智能体 → MCP 协议 → Skills 技能
大脑 → 图书馆 → 手和脚 → USB 接口 → App Store

一、大模型基础：AI 的「大脑」

在理解所有上层应用之前，必须先理解基座——大语言模型（Large Language Model，LLM）。

把大模型想象成一个读了全世界所有书的超级学霸。他读过维基百科的每篇文章、GitHub 上的每行代码、Reddit 上的每条帖子。知识面广到不可思议。但他有个关键特点：不会上网搜索，只能凭记忆回答。记忆模糊时，他甚至会自信满满地编一个答案——这就是幻觉（Hallucination）的由来。

Token 与上下文窗口

大模型不是逐字阅读文本的，它把文本切成小块——Token（词元），这是 AI 的阅读单位。一个汉字通常 1-2 个 Token，一个英文单词 1-3 个 Token。模型的计费、速度、处理能力都以 Token 为单位。

上下文窗口（Context Window）决定模型一次能看到多少内容——如同人的工作记忆。GPT-4 Turbo 有 128K Token 窗口（约 10 万字），Claude 支持最高 200K Token。窗口越大，能处理的文本越长。

为什么上下文窗口很重要？

想象你做一道数学大题，但只能同时看到题目前两行——很难把握全貌。上下文窗口就是模型的视野。但代价是：窗口越大，计算量越大，推理越慢，成本越高。这是一个核心的工程权衡。

当你向 AI 提问时，模型做的事叫推理（Inference）——逐词预测最可能的下一个词。今天天气真 → 好。看似简单，背后是每秒数十亿次浮点运算。

Transformer 架构：大模型的基石

2017 年，Google 发表了 Attention Is All You Need，提出 Transformer 架构。核心创新是自注意力机制（Self-Attention）——模型处理每个词时，能关注到输入中所有其他词的信息，而非像传统 RNN 只能顺序处理。

举例：小明把苹果给了小红，因为她饿了——模型需要理解她指小红。自注意力让模型能直接建立她和小红之间的关联，不管隔多远。这种全局关注是 Transformer 超越此前架构的关键。

另一特性是可并行化：传统 RNN 必须逐词处理，Transformer 同时处理整个序列，能充分利用 GPU 并行能力，扩展到数千亿参数。

训练三部曲：从白纸到专家

第一步：预训练（Pre-training）——用万亿级 Token 训练预测下一个词，获得广泛知识。类比：一个人读了所有书，什么都知道但不会好好交流。成本极高：数千张 GPU 训练数月。

第二步：指令微调（SFT）——用标注好的问-答数据教模型以回答格式响应。经过 SFT，从语言模型变成助手。

第三步：RLHF——让人类对多个回答排序，训练奖励模型，再通过强化学习优化。ChatGPT 从聪明但不好用变成既聪明又好用的关键。

预训练
海量数据 → 知识 → SFT
标注数据 → 对话 → RLHF
人类反馈 → 质量

二、RAG：给 AI 配一个「图书馆」

大模型虽然博学，但有三个致命缺陷：知识有截止日期、不含私有信息、会产生幻觉。RAG（Retrieval-Augmented Generation，检索增强生成）就是为解决这些问题。

把 RAG 想象成开卷考试——不需要背下所有内容，只需知道去哪找、怎么用。RAG 让 AI 在回答前先查资料，基于真实文档生成回答。

为什么需要 RAG

  • 知识时效性：模型的训练数据有截止日期，RAG 让 AI 检索最新数据。
  • 企业私有知识：公司内部文档模型从未见过，RAG 让 AI 使用你的私有数据。
  • 减少幻觉：基于真实文档回答，大幅降低一本正经胡说八道的概率，且回答可溯源。

向量化与向量数据库

RAG 核心技术是向量化（Vector Embedding）——把文本转换成一组数字（向量），让语义相近的文本在数学空间中彼此靠近。

想象一个巨大坐标系。小狗在一个位置，柯基金毛在附近，汽车在很远处。传统搜索只能关键词匹配——搜小狗找不到柯基。向量搜索能理解语义，知道柯基就是一种小狗。

技术上，向量化通过 Embedding 模型（如 OpenAI 的 text-embedding-ada-002）实现，把文本转为高维向量（通常 1536 维浮点数组）。向量数据库（Pinecone、Milvus、Qdrant、pgvector）专门存储和检索这些向量，使用 HNSW、IVF 等索引算法实现毫秒级相似性搜索。

RAG 的完整检索流程

用户提问 → 问题向量化 → 向量检索 → 拼接上下文 → LLM 生成

离线阶段：文档切分成小段（Chunk），每段用 Embedding 模型转为向量存入数据库。在线阶段：用户提问 → 问题向量化 → 检索 top-k 相关段落 → 段落+问题拼成 Prompt → 大模型生成回答。

rag_pseudocode.py
# RAG 检索流程伪代码 def rag_query(question, vector_db, llm): query_vec = embed(question) # 问题向量化 docs = vector_db.search(query_vec, top_k=5) # 检索 context = "\n".join([d.text for d in docs]) prompt = f"基于以下资料回答：\n{context}\n\n问题：{question}" return llm.generate(prompt)
RAG 的优缺点

优点：知识可实时更新、支持私有数据、回答可溯源、降低幻觉。缺点：检索质量依赖切分策略和 Embedding 模型、增加系统复杂度和延迟。实际工程中，切分策略（Chunking Strategy）往往决定 80% 的效果。

三、Agent：AI 有了「手」和「脚」

如果大模型是「大脑」，RAG 是「图书馆」，那么 Agent 就是让 AI 拥有「手」和「脚」——从"能聊天"到"能干活"的跃迁。

普通聊天机器人只能对话——你问天气，它说"我无法查看实时天气"。Agent 不同：你问天气，它自己调用天气 API 后告诉你结果；你让它订机票，它自己搜索、比价、填表单。Agent 不只是说，它还能做。

感知-规划-执行-反馈循环

👁 感知 Perceive → 🎯 规划 Plan → 🔧 执行 Act → 📊 反馈 Reflect → (循环)
  • 感知：接收用户指令、环境信息、工具返回结果
  • 规划：分析任务，分解步骤，制定策略（大模型的核心价值）
  • 执行：调用工具、运行代码、操作外部系统
  • 反馈：评估结果，判断是否完成，决定是否继续

循环可能只运行一次（简单任务），也可能数十次（复杂任务）。Agent 的强大在于能自主决定调什么工具、什么顺序、何时停下——这和传统 if-else 程序截然不同。

工具调用（Function Calling）

Agent 之所以能"干活"，核心是工具调用。大模型本身不能上网、不能读文件——但我们给它一份「工具菜单」。当模型判断需要某工具时，输出结构化调用请求，由外部系统执行后返回结果。

function_calling_demo.json
// 给模型的工具描述 { "name": "get_weather", "description": "查询指定城市的实时天气", "parameters": {"city": "string", "date": "string"} } // 对话过程 用户：帮我查北京今天的天气 Agent 思考：需要调用 get_weather → 调用：get_weather(city="北京", date="today") ← 返回：{"temp": "22°C", "weather": "晴"} Agent：北京今天晴朗 22°C，适合外出！

关键：模型只负责决策，实际执行由外部系统完成。这既保证灵活性，也保证安全性——敏感操作可在外部加权限控制。

四、MCP 协议：AI 世界的「USB 接口」

Function Calling 让 AI 能用工具，但每个 AI 应用都要为每个工具写一套对接代码。MCP（Model Context Protocol）由 Anthropic 提出，是开放标准协议，让所有 AI 应用和工具之间只需一种"接口"。

为什么需要统一协议

USB 出现前，打印机一种接口、键盘一种、鼠标又一种。USB 统一了这一切。MCP 做同样的事：所有工具遵循同一套协议。AI 应用只需实现一次 MCP 客户端，就能连接任何 MCP 服务器的工具。

没有 MCP：10 应用 × 10 工具 = 100 套代码。有了 MCP：10 + 10 = 20 套。生态越大，优势越明显。

三大核心概念

Tools（工具）：可被 AI 主动调用的函数——"搜索文件""执行 SQL""发邮件"。每个工具有明确输入参数和输出格式。

Resources（资源）：AI 可读取的数据源——文件内容、数据库记录、API 返回。资源是只读的。类比：Tools 是"手"，Resources 是"眼睛"。

Prompts（提示模板）：预定义的交互模板，如"代码审查"模板，让 AI 按特定方式处理特定任务。

MCP 实际应用

GitHub MCP：创建 Issue、提交 PR。数据库 MCP：查询 SQL、分析数据。文件系统 MCP：读写文件。Slack MCP：发消息、搜聊天记录。社区已有数百个 MCP Server 可用。

五、Skills 技能系统：AI 的「App Store」

如果 MCP 是"接口标准"（USB 协议），那么 Skills 是更高层的能力封装（App Store 里的应用）。一个 Skill 不仅包含工具调用，还包含指令、脚本、资源文件和上下文知识——完整的"能力包"。

想让 Agent 学会分析股票？安装"股票分析"技能——包含 K 线图脚本、技术指标工具、财报分析模板。安装后只需说"帮我分析茅台走势"，它就知道该怎么做。技能把一切打包好了。

Skills 理念：让"安装一个能力"像安装 App 一样简单。未来的 AI 助手生态，很可能就是"基座 Agent + 按需安装 Skills"。

六、Agent 记忆系统：让 AI「记住」你

Agent 不只是一次性的"问-答"。它需要记住你是谁、你的偏好、之前的对话，才能像真正的助手那样工作。

短期记忆（Working Memory）——就是当前对话的上下文窗口。Agent 记得你这轮对话说过的每句话。但关闭对话后，这些记忆就消失——就像人挂了电话就忘了刚才说的号码。受限于上下文窗口大小。

长期记忆（Long-term Memory）——通过文件系统、数据库或向量存储持久保存信息。Agent 可以记住你的工作习惯、项目进展、个人偏好。即使关闭会话，下次对话时它仍然"记得"你。

短期记忆像白板——当前讨论写在上面，擦了就没了。长期记忆像笔记本——重要的东西记下来，什么时候都能翻看。优秀的 Agent 需要两者配合：用短期记忆理解当前任务，用长期记忆提供个性化服务。

技术实现上，长期记忆通常有三种方案：

  • 文件存储：最简单直接，把记忆写成 Markdown 文件。优点是可读可编辑，缺点是检索效率低。
  • 向量存储：把记忆向量化后存入向量数据库，检索时通过语义相似度匹配。适合大量非结构化记忆。
  • 结构化数据库：用传统数据库存储结构化信息（用户偏好、任务历史等）。查询精确但灵活性差。

实际产品中，往往是多种方案的组合。比如 OpenClaw 使用文件系统存储日常记忆（memory/YYYY-MM-DD.md），同时用一个长期记忆文件（MEMORY.md）保存精炼后的关键信息——就像人类的日记和长期记忆的关系。

七、多 Agent 协作：一个人干不了所有事

一个人干不了所有事，一个 Agent 也是。复杂任务需要多个专业 Agent 分工合作——就像一个团队。

分工模式：一个"程序员 Agent"负责写代码，一个"测试 Agent"负责找 Bug，一个"产品经理 Agent"负责审查需求。每个 Agent 专注于自己擅长的领域，协作完成复杂项目。

编排与调度：需要一个"指挥官"来协调各个 Agent——谁先干、谁后干、谁的结果传给谁。这就是 Agent 编排（Orchestration）。编排可以是固定流程（像流水线），也可以让 AI 动态决定（像项目经理临场调度）。

编排器

↓

多 Agent 协作的挑战在于通信效率和一致性。Agent 之间需要共享上下文，但又不能把所有信息都传给每个 Agent（会超出上下文窗口）。如何设计高效的信息传递机制，是多 Agent 系统的核心难题。

八、产品图鉴：谁在做 Agent？

从编程助手到通用 Agent，这些产品正在定义 AI 应用的未来。

Claude Code（Anthropic）——终端原生的 AI 编程助手。在命令行中直接与 AI 协作，能自主规划、编写、测试、调试。不只是代码补全，而是完整的 Agent 模式。原生支持 MCP 协议。适合开发者和技术团队。

Cursor——AI 原生代码编辑器。基于 VS Code 深度改造，内置多模型支持，Tab 智能补全 + Chat 对话编程 + Composer 多文件协同。把 AI 深度融入编辑器体验，学习成本最低。适合所有开发者。

Dify——开源 LLM 应用开发平台。可视化工作流编排，拖拽式构建 AI 应用。内置 RAG 引擎，Agent 构建器。不写代码也能搭建 AI 应用，15 分钟构建一个客服机器人。适合产品经理和运营团队。

n8n——开源工作流自动化平台。400+ 预置集成节点，AI Agent 节点让 AI 做工作流中的决策者。支持自托管，数据完全掌控。连接邮件、飞书、数据库、API，让 AI 在工作流中做智能决策。

Manus——通用型 AI Agent。能上网浏览、搜索信息、写代码、运行程序。自主规划多步骤任务，给它一个目标，它自己规划路径并执行。像一个远程助理，你只需说"做什么"。

OpenClaw——个人 AI Agent 运行时框架。多渠道支持（微信、QQ、飞书、Discord、Telegram），技能系统按需安装，原生 MCP 支持，完全本地部署。你的 Agent 运行在你自己的服务器上——真正属于你的 AI 助手。

九、未来展望

每个人的数字分身——想象你有一个 AI Agent 完全了解你的工作习惯、知识偏好、社交风格。它帮你筛选邮件、整理日程、撰写报告、管理项目。它不只是工具，更像你的数字化延伸——一个 7×24 小时不休息的"你"。这不是科幻，这是正在发生的未来。

从工具到伙伴——当前的 AI 还是"工具"：你给指令，它执行。未来的 Agent 将更像"伙伴"——它理解你的目标和价值观，能在你没想到的时候主动帮助。"我注意到你这周压力很大，要不要帮你推迟几个非紧急会议？"这种主动的、有同理心的协作，是 Agent 进化的方向。

安全、隐私、可控——Agent 越强大，风险越大。当 AI 能自主操作你的电脑、访问你的数据、代表你发消息时，安全和隐私就是头等大事。谁来监督 Agent 的行为？敏感操作是否需要人类确认？数据存储在哪里？这些问题的答案，将决定我们能走多远。

写在最后

从大模型到 Agent，我们正在经历 AI 应用从"能聊天"到"能干活"的范式转变。这个技术栈还在快速演进——今天的最佳实践可能明天就被颠覆。但底层逻辑是清晰的：大模型提供智能，RAG 提供知识，Agent 提供行动力，MCP 提供连接性，Skills 提供可扩展性。理解了这条线索，你就能在这个快速变化的领域中保持方向感。
